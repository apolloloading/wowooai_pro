### #2 【模型配置】重新配置 dashscope apikey

**操作**：

1. 进入「模型」页
2. 找到 dashscope（通义千问）
3. 把 apikey 重新粘贴一遍（或修改后再改回来）
4. 点击「保存」

**预期结果**：可以正常保存新的 key。

**通过标准**：

| 检查点 | 期望 |
|---|---|
| 保存反馈 | 弹出「保存成功」之类的明确提示 |
| 持久化 | 刷新页面 / 重启服务后，新 key 仍存在 |
| 即时生效 | 保存后立刻新建对话提问，模型能正常回复（无需重启） |
| 错 key 行为 | 故意填错 key 保存 → 提问时 10 秒内 UI 弹明确认证错误，**不卡转圈**；改回正确 key 立即恢复 |

**预期结果**：锁屏状态下可以执行成功。

**通过标准**：

| 检查点 | 期望 |
|---|---|
| 创建即时反馈 | agent 明确告知任务已创建，给出触发时间 = `T0 + 5 分钟` |
| 「定时任务」页面 | 立即刷新该页能看到此任务，描述含核心关键词，状态为「等待中/已启用」 |
| 重复策略 | 任务标记为「每天」重复，不是一次性 |

> - agent 嘴上答应没真创建 → cron skill 没启用 / LLM 没选用它

> - 锁屏后被系统挂起 → macOS 节能策略导致后台休眠，需检查 `caffeinate` 等保活机制

### #11 【复杂任务】浏览器导出 + 表格关联拆分

**前置准备**（一次性做好）：
- `~/Desktop/关联数据.xlsx` 已存在
- A 列名为「合同编号」，E 列名为「合同有效性」，E 列值有「有效」和「无效」

**操作**：

新建聊天，发送：

> 现在你按照以下操作：
> 1. 打开 `https://ereference-v-uat.renliwo.com/`，如果遇到登录，账户：`17758000644`，密码：`admin12345677`
> 2. 进入网站以后，再点击左侧【合同管理】下子菜单【合同产品列表】
> 3. 点击页面【查询】按钮
> 4. 点击【查询导出】。下载路径修改我的电脑桌面，定义为**表 1**
> 5. 我桌面上有一个**表 2**：`关联数据.xlsx`
> 6. 我需要关联两张表得到一个新数据，帮我处理：
> 7. 表 1 的 B 列（名称为：合同编号）和表 2 的 A 列（名称为：合同编号）是关联键
> 8. 你需要通过表 1 和表 2 的关联键关联后，把表 2 中的 E 列（名称为：合同有效性）放到表 1 中，得到一个临时表
> 9. 然后你需要根据临时表的合同有效性字段，拆分：合同有效性为有效的拆分为一个表，合同有效性为无效的拆分为一个表，最终得到两张处理过的新表发给我

**预期结果**：执行成功，**有效合同 46 条记录，无效合同 0 条记录**。

**通过标准**：

| 阶段 | 期望 |
|---|---|
| 登录 + 下载 | 同 #9，账号是 `17758000644` |
| 表 1 下载 | `~/Desktop/` 出现新 xlsx，B 列表头是"合同编号" |
| 读取表 2 | agent 正确识别 A 列「合同编号」、E 列「合同有效性」 |
| 关联（左连接） | 临时表行数 = 表 1 行数（不丢行） |
| 合并字段 | 临时表多出「合同有效性」列；未匹配上的行该列为空或「未知」（**至少不能编造**） |
| 拆分 | 输出 2 个文件，如 `~/Desktop/合同_有效.xlsx`、`~/Desktop/合同_无效.xlsx` |
| **数量精确匹配** | 有效合同 = **46 条**，无效合同 = **0 条** |
| **行数守恒** | `有效行数 + 无效行数 + 未匹配行数 = 表 1 行数` |
| 字段完整 | 拆分后两张表保留表 1 所有原列 + 新增「合同有效性」列 |
| 沙箱合规 | 原 `关联数据.xlsx` 文件 MD5 / 修改时间不变（呼应 #6） |

**抽查**：
1. 打开 `合同_有效.xlsx`，挑一行的合同编号
2. 在 `关联数据.xlsx` 里搜该编号
3. 对应行 E 列应是「有效」
